# Custom Composables

Composables are functions that use Vue's Composition API to encapsulate and reuse **stateful logic**. They are the primary reusability pattern in Vue 3, replacing mixins.

## What is a Composable?

A composable is a function whose name starts with `use` and leverages Vue reactivity internally:

```ts
// composables/useCounter.ts
import { ref } from 'vue'

export function useCounter(initial = 0) {
  const count = ref(initial)

  function increment() {
    count.value++
  }

  function decrement() {
    count.value--
  }

  function reset() {
    count.value = initial
  }

  return { count, increment, decrement, reset }
}
```

Usage in a component:

```vue
<script setup lang="ts">
import { useCounter } from '~/composables/useCounter'

const { count, increment, decrement, reset } = useCounter(10)
</script>

<template>
  <div>
    <p>Count: {{ count }}</p>
    <button @click="increment">+</button>
    <button @click="decrement">-</button>
    <button @click="reset">Reset</button>
  </div>
</template>
```

## Real-World Example: useMousePosition

A composable that tracks mouse position and cleans up on unmount:

```ts
// composables/useMousePosition.ts
import { ref, onMounted, onUnmounted } from 'vue'

export function useMousePosition() {
  const x = ref(0)
  const y = ref(0)

  function update(event: MouseEvent) {
    x.value = event.pageX
    y.value = event.pageY
  }

  onMounted(() => window.addEventListener('mousemove', update))
  onUnmounted(() => window.removeEventListener('mousemove', update))

  return { x, y }
}
```

This demonstrates the key advantage: **the lifecycle cleanup is co-located with the setup**, unlike mixins where setup and teardown are scattered across different hooks.

## Async Composables

Composables can handle async operations with loading and error states:

```ts
// composables/useFetchData.ts
import { ref } from 'vue'

export function useFetchData<T>(url: string) {
  const data = ref<T | null>(null)
  const error = ref<string | null>(null)
  const loading = ref(false)

  async function execute() {
    loading.value = true
    error.value = null
    try {
      const response = await fetch(url)
      if (!response.ok) throw new Error(`HTTP ${response.status}`)
      data.value = await response.json()
    } catch (e) {
      error.value = e instanceof Error ? e.message : 'Unknown error'
    } finally {
      loading.value = false
    }
  }

  return { data, error, loading, execute }
}
```

```vue
<script setup lang="ts">
const { data, error, loading, execute } = useFetchData<User[]>('/api/users')

// Fetch on mount
onMounted(execute)
</script>

<template>
  <div v-if="loading">Loading...</div>
  <div v-else-if="error">Error: {{ error }}</div>
  <ul v-else-if="data">
    <li v-for="user in data" :key="user.id">{{ user.name }}</li>
  </ul>
</template>
```

## Composable Conventions

### Naming

- Always prefix with `use` — `useCounter`, `useAuth`, `useDarkMode`
- Name describes the **capability**, not the implementation

### Input / Output

- Accept refs **or** plain values as input (use `toValue()` or `unref()` internally)
- Always return **refs**, not reactive objects — callers can destructure safely

```ts
import { toValue, type MaybeRefOrGetter } from 'vue'

export function useDoubled(value: MaybeRefOrGetter<number>) {
  return computed(() => toValue(value) * 2)
}

// Works with ref, getter, or plain value:
useDoubled(count)       // ref
useDoubled(() => count) // getter
useDoubled(5)           // plain value
```

### Return Object vs Tuple

```ts
// Object return (most common) — easy to destructure selectively
return { x, y, isOutside }

// Tuple return (rare) — useful for simple 2-value returns
return [x, y] as const
```

## Composables vs Mixins

| Feature | Composables | Mixins |
|---|---|---|
| Name collisions | No (explicit destructuring) | Yes (implicit merge) |
| Data source clarity | Clear (you see the import) | Unclear ("where did this come from?") |
| TypeScript support | Full | Poor |
| Lifecycle co-location | Yes | No (scattered across hooks) |
| Flexible params | Yes (function arguments) | Limited |

## Best Practices

1. **One responsibility per composable** — `useAuth`, not `useAuthAndThemeAndRouter`
2. **Co-locate side effects with cleanup** — `onMounted` + `onUnmounted` together
3. **Return refs, accept MaybeRefOrGetter** — maximum flexibility
4. **Keep composables pure when possible** — side effects only when necessary
5. **In Nuxt, put composables in `composables/`** — they are auto-imported

## Summary

- Composables are `use`-prefixed functions encapsulating reactive logic
- They replace mixins with better DX, TypeScript support, and explicit data flow
- Accept flexible input (`MaybeRefOrGetter`), return refs
- Co-locate setup and cleanup for clean lifecycle management
