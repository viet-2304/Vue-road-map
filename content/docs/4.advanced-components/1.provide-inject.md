# Provide / Inject

When building component trees in Vue, you often need to pass data from a parent component to a deeply nested child. Doing this through props alone means every intermediate component must declare and forward those props -- a problem known as **prop drilling**. Vue's `provide` and `inject` mechanism solves this elegantly.

## The Prop Drilling Problem

Consider a component tree like this:

```
App
 └── Dashboard
      └── Sidebar
           └── UserAvatar
```

If `App` holds the user data and `UserAvatar` needs it, you would have to pass the `user` prop through `Dashboard` and `Sidebar`, even though they do not use it themselves. This creates tight coupling and maintenance headaches.

## Basic Provide and Inject

With the Composition API, a parent component can **provide** a value, and any descendant can **inject** it -- no matter how deep.

### Providing a Value

```vue
<script setup>
import { provide } from 'vue'

const theme = 'dark'

provide('theme', theme)
</script>
```

The first argument is the **injection key** (a string or Symbol), and the second is the value to provide.

### Injecting a Value

```vue
<script setup>
import { inject } from 'vue'

const theme = inject('theme')
</script>

<template>
  <div :class="`panel-${theme}`">
    Themed content
  </div>
</template>
```

Any descendant of the providing component can call `inject` with the same key to retrieve the value.

## Reactive Provide / Inject

A plain string or number provided via `provide` is **not reactive**. If the parent changes the value, the child will not update. To make provide/inject reactive, provide a `ref` or `reactive` object.

```vue
<!-- Parent.vue -->
<script setup>
import { ref, provide } from 'vue'

const count = ref(0)

provide('count', count)

function increment() {
  count.value++
}
</script>

<template>
  <button @click="increment">Increment</button>
  <slot />
</template>
```

```vue
<!-- DeepChild.vue -->
<script setup>
import { inject } from 'vue'

const count = inject('count')
</script>

<template>
  <p>Count from ancestor: {{ count }}</p>
</template>
```

Because `count` is a `ref`, any change in the parent automatically updates every component that injected it.

### Providing Computed Values

You can also provide `computed` properties for derived reactive state:

```vue
<script setup>
import { ref, computed, provide } from 'vue'

const items = ref([1, 2, 3])
const total = computed(() => items.value.reduce((a, b) => a + b, 0))

provide('total', total)
</script>
```

## Keeping Mutations Predictable

When you provide a reactive value, any descendant could mutate it directly. This makes debugging difficult. The recommended pattern is to provide both the value and a mutation function, and keep the mutation logic in the provider.

```vue
<!-- Provider.vue -->
<script setup>
import { ref, provide, readonly } from 'vue'

const count = ref(0)

function increment() {
  count.value++
}

provide('counter', {
  count: readonly(count),
  increment
})
</script>
```

```vue
<!-- Consumer.vue -->
<script setup>
import { inject } from 'vue'

const { count, increment } = inject('counter')
// count is readonly -- direct mutation will warn in dev mode
</script>

<template>
  <p>{{ count }}</p>
  <button @click="increment">+1</button>
</template>
```

Using `readonly()` prevents the child from mutating the ref directly, enforcing a one-way data flow.

## Default Values

If a component calls `inject` but no ancestor provides the matching key, the result is `undefined` by default. You can supply a fallback as the second argument:

```vue
<script setup>
import { inject } from 'vue'

const theme = inject('theme', 'light')
</script>
```

For expensive defaults, pass a factory function as the third argument to avoid unnecessary computation:

```vue
<script setup>
import { inject } from 'vue'

const config = inject('config', () => {
  return buildExpensiveDefaultConfig()
}, true) // third arg `true` marks it as a factory
</script>
```

The `true` flag tells Vue to treat the function as a factory rather than the literal default value.

## Symbol Keys

String keys work fine in small apps, but in larger codebases or libraries, they risk collisions. Using `Symbol` keys guarantees uniqueness.

```ts
// keys.ts
import type { InjectionKey, Ref } from 'vue'

export const CounterKey: InjectionKey<Ref<number>> = Symbol('counter')
```

```vue
<!-- Provider.vue -->
<script setup lang="ts">
import { ref, provide } from 'vue'
import { CounterKey } from './keys'

const count = ref(0)
provide(CounterKey, count)
</script>
```

```vue
<!-- Consumer.vue -->
<script setup lang="ts">
import { inject } from 'vue'
import { CounterKey } from './keys'

const count = inject(CounterKey) // TypeScript knows this is Ref<number> | undefined
</script>
```

`InjectionKey` is a special type from Vue that links the key to its value type, giving you full type safety at the inject site.

## App-Level Provide

You can provide values at the app level so they are available to every component in the entire application. This is useful for global configuration, themes, or service instances.

```ts
// main.ts
import { createApp } from 'vue'
import App from './App.vue'

const app = createApp(App)

app.provide('apiBaseUrl', 'https://api.example.com')
app.provide('featureFlags', {
  darkMode: true,
  betaFeatures: false
})

app.mount('#app')
```

Any component in the app can inject these values without a specific parent providing them.

## Building a Simple Theme System

Here is a practical example that ties the concepts together:

```ts
// theme.ts
import type { InjectionKey, Ref } from 'vue'

export interface Theme {
  primary: string
  background: string
  text: string
}

export const ThemeKey: InjectionKey<{
  theme: Readonly<Ref<Theme>>
  setTheme: (t: Theme) => void
}> = Symbol('theme')
```

```vue
<!-- ThemeProvider.vue -->
<script setup lang="ts">
import { ref, provide, readonly } from 'vue'
import { ThemeKey, type Theme } from './theme'

const theme = ref<Theme>({
  primary: '#3b82f6',
  background: '#ffffff',
  text: '#1f2937'
})

function setTheme(newTheme: Theme) {
  theme.value = newTheme
}

provide(ThemeKey, {
  theme: readonly(theme),
  setTheme
})
</script>

<template>
  <slot />
</template>
```

```vue
<!-- ThemedButton.vue -->
<script setup lang="ts">
import { inject } from 'vue'
import { ThemeKey } from './theme'

const themeCtx = inject(ThemeKey)

if (!themeCtx) {
  throw new Error('ThemedButton must be used inside a ThemeProvider')
}

const { theme } = themeCtx
</script>

<template>
  <button :style="{ backgroundColor: theme.primary, color: theme.text }">
    <slot />
  </button>
</template>
```

## Common Pitfalls

1. **Forgetting reactivity.** If you provide a plain value, changes will not propagate. Always provide a `ref` or `reactive` object when the value can change.

2. **Mutating injected values directly.** Even though it works, it makes the data flow hard to trace. Prefer `readonly` plus an explicit update function.

3. **Missing providers.** If no ancestor provides the key and you forget a default, `inject` returns `undefined`. This can cause silent runtime errors. Always provide a default or add a runtime check.

4. **Overuse.** Provide/inject is not a replacement for a state management solution. Use it for dependency injection patterns (themes, configs, service instances), not for complex shared state. For that, consider Pinia.

## Summary

- `provide` and `inject` let you share data across a component tree without prop drilling.
- Provide `ref` or `reactive` values to keep injected data reactive.
- Use `readonly` to enforce one-way data flow.
- Use `Symbol` keys with `InjectionKey` for type safety and collision avoidance.
- Supply default values to guard against missing providers.
- App-level `provide` makes values available globally.
