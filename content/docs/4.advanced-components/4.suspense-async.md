# Suspense & Async Components

As applications grow, loading every component upfront becomes expensive. Vue provides `defineAsyncComponent` to load components on demand and the `<Suspense>` built-in component to orchestrate loading states. Together, they give you fine-grained control over the user experience during asynchronous operations.

## defineAsyncComponent

The `defineAsyncComponent` function creates a component that is loaded lazily -- only when it is actually needed.

### Basic Usage

```js
import { defineAsyncComponent } from 'vue'

const AsyncChart = defineAsyncComponent(() =>
  import('./components/HeavyChart.vue')
)
```

Vue will not load `HeavyChart.vue` until `AsyncChart` is rendered for the first time. The import returns a Promise, and Vue handles the rest.

You use async components just like regular ones:

```vue
<template>
  <AsyncChart v-if="showChart" :data="chartData" />
</template>
```

### Advanced Options

`defineAsyncComponent` accepts an options object for more control:

```js
import { defineAsyncComponent } from 'vue'
import LoadingSpinner from './LoadingSpinner.vue'
import ErrorDisplay from './ErrorDisplay.vue'

const AsyncDashboard = defineAsyncComponent({
  loader: () => import('./Dashboard.vue'),
  loadingComponent: LoadingSpinner,
  errorComponent: ErrorDisplay,
  delay: 200,       // ms before showing loading component
  timeout: 10000    // ms before triggering error state
})
```

| Option | Description |
|---|---|
| `loader` | Function returning a Promise (the dynamic import) |
| `loadingComponent` | Component shown while loading |
| `errorComponent` | Component shown if the load fails |
| `delay` | Delay in ms before showing the loading component (default: 200) |
| `timeout` | Time in ms after which loading is considered failed |

The `delay` option prevents a flash of loading state for fast connections. If the component loads within 200ms, the loading spinner never appears.

### Error Component Props

The error component receives useful props you can use:

```vue
<!-- ErrorDisplay.vue -->
<script setup>
const props = defineProps({
  error: Error,
  retry: Function
})
</script>

<template>
  <div class="error-panel">
    <p>Failed to load: {{ error.message }}</p>
    <button @click="retry">Try Again</button>
  </div>
</template>
```

The `retry` function lets users attempt to reload the failed component.

## The Suspense Component

`<Suspense>` is a built-in component that handles async dependencies in your component tree. It renders **fallback** content while waiting for async components or async `setup()` functions to resolve.

### Basic Suspense

```vue
<template>
  <Suspense>
    <template #default>
      <AsyncDashboard />
    </template>
    <template #fallback>
      <div class="loading">Loading dashboard...</div>
    </template>
  </Suspense>
</template>
```

`<Suspense>` has two slots:

- **`#default`** -- the actual content to render (may contain async components).
- **`#fallback`** -- shown while the default content is resolving.

### Async Setup

`<Suspense>` also works with components that have an async `setup()` function. When using `<script setup>`, a top-level `await` makes the setup async automatically:

```vue
<!-- UserProfile.vue -->
<script setup>
const response = await fetch('/api/user/1')
const user = await response.json()
</script>

<template>
  <div class="profile">
    <h2>{{ user.name }}</h2>
    <p>{{ user.email }}</p>
  </div>
</template>
```

This component will not render until the fetch completes. Wrap it in `<Suspense>` to show a fallback during the wait:

```vue
<template>
  <Suspense>
    <UserProfile />
    <template #fallback>
      <p>Loading profile...</p>
    </template>
  </Suspense>
</template>
```

## Suspense Events

`<Suspense>` emits events that let you track its state:

```vue
<template>
  <Suspense
    @pending="onPending"
    @resolve="onResolve"
    @fallback="onFallback"
  >
    <AsyncContent />
    <template #fallback>
      <LoadingSpinner />
    </template>
  </Suspense>
</template>

<script setup>
function onPending() {
  console.log('Async dependencies detected, waiting...')
}

function onResolve() {
  console.log('All async dependencies resolved!')
}

function onFallback() {
  console.log('Fallback content is now displayed')
}
</script>
```

| Event | When it fires |
|---|---|
| `pending` | When entering a pending state (new async deps detected) |
| `resolve` | When all async dependencies have resolved |
| `fallback` | When the fallback content is shown |

## Error Handling with onErrorCaptured

`<Suspense>` does not have built-in error UI. You need to handle errors in the parent component using the `onErrorCaptured` lifecycle hook:

```vue
<script setup>
import { ref, onErrorCaptured } from 'vue'

const error = ref(null)

onErrorCaptured((err) => {
  error.value = err
  return false // prevents further propagation
})
</script>

<template>
  <div v-if="error" class="error-banner">
    <p>Something went wrong: {{ error.message }}</p>
    <button @click="error = null">Dismiss</button>
  </div>

  <Suspense v-else>
    <AsyncDashboard />
    <template #fallback>
      <p>Loading...</p>
    </template>
  </Suspense>
</template>
```

This pattern creates an **error boundary** -- a component that catches errors from its descendants and displays a fallback UI.

### Reusable Error Boundary Component

You can extract this into a reusable component:

```vue
<!-- ErrorBoundary.vue -->
<script setup>
import { ref, onErrorCaptured } from 'vue'

const error = ref(null)

onErrorCaptured((err) => {
  error.value = err
  return false
})

function reset() {
  error.value = null
}
</script>

<template>
  <slot v-if="!error" />
  <slot v-else name="error" :error="error" :reset="reset">
    <div class="error-fallback">
      <p>An error occurred: {{ error.message }}</p>
      <button @click="reset">Retry</button>
    </div>
  </slot>
</template>
```

```vue
<!-- Usage -->
<template>
  <ErrorBoundary>
    <Suspense>
      <AsyncWidget />
      <template #fallback>
        <p>Loading widget...</p>
      </template>
    </Suspense>

    <template #error="{ error, reset }">
      <div class="custom-error">
        <p>Widget failed: {{ error.message }}</p>
        <button @click="reset">Try Again</button>
      </div>
    </template>
  </ErrorBoundary>
</template>
```

## Nested Suspense

You can nest `<Suspense>` boundaries to control loading states at different levels:

```vue
<template>
  <Suspense>
    <template #default>
      <div class="dashboard">
        <DashboardHeader />

        <div class="widgets">
          <Suspense>
            <AsyncChartWidget />
            <template #fallback>
              <div class="skeleton chart-skeleton" />
            </template>
          </Suspense>

          <Suspense>
            <AsyncTableWidget />
            <template #fallback>
              <div class="skeleton table-skeleton" />
            </template>
          </Suspense>
        </div>
      </div>
    </template>

    <template #fallback>
      <p>Loading entire dashboard...</p>
    </template>
  </Suspense>
</template>
```

The outer `<Suspense>` waits for `DashboardHeader` if it is async. The inner `<Suspense>` boundaries let each widget load independently with its own skeleton placeholder. This creates a progressive loading experience.

## Combining with Transitions

`<Suspense>` works with `<Transition>` to animate the swap between fallback and resolved content:

```vue
<template>
  <RouterView v-slot="{ Component }">
    <Suspense>
      <template #default>
        <Transition name="fade" mode="out-in">
          <component :is="Component" />
        </Transition>
      </template>
      <template #fallback>
        <LoadingBar />
      </template>
    </Suspense>
  </RouterView>
</template>
```

## Code Splitting with Vue Router

Async components pair naturally with Vue Router for route-level code splitting:

```js
// router.js
const routes = [
  {
    path: '/',
    component: () => import('./pages/Home.vue')
  },
  {
    path: '/dashboard',
    component: () => import('./pages/Dashboard.vue')
  },
  {
    path: '/settings',
    component: () => import('./pages/Settings.vue')
  }
]
```

Each route component is loaded only when the user navigates to it. The bundler (Vite/webpack) automatically creates separate chunks for each.

## Common Pitfalls

1. **Suspense is experimental.** As of Vue 3.5, `<Suspense>` is still marked as experimental. Its API may change in future releases. Use it with awareness of this status.

2. **Top-level await blocks rendering.** A component with `await` in `<script setup>` will not render until all awaits resolve. Keep async setup logic minimal; move heavy async work to `onMounted` when you want the component to render immediately.

3. **Error handling is not automatic.** Unlike React's error boundaries, Vue's `<Suspense>` does not catch errors by default. You must use `onErrorCaptured` in a parent component.

4. **Hydration with SSR.** Async components behave differently during server-side rendering. In Nuxt, use `<LazyComponent>` prefix (auto-generated by Nuxt) or handle async components carefully to avoid hydration mismatches.

5. **Multiple async roots.** `<Suspense>` waits for ALL async dependencies in its default slot. If one component is slow, everything waits. Use nested `<Suspense>` to allow independent loading.

## Summary

- `defineAsyncComponent` enables lazy loading of components with built-in loading and error states.
- `<Suspense>` orchestrates fallback content while async dependencies resolve.
- Components with top-level `await` in `<script setup>` are automatically async.
- Use `onErrorCaptured` to build error boundaries around `<Suspense>`.
- Nest `<Suspense>` boundaries for independent, progressive loading.
- Combine with Vue Router for route-level code splitting.
