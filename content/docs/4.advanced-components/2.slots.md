# Slots

Slots are one of Vue's most powerful content distribution mechanisms. They allow a parent component to inject template content into a child component's layout, making components truly flexible and reusable.

## Default Slot

The simplest form of a slot is the **default slot**. A child component declares where parent content should appear using the `<slot>` element.

```vue
<!-- Card.vue -->
<template>
  <div class="card">
    <slot />
  </div>
</template>
```

```vue
<!-- Parent.vue -->
<template>
  <Card>
    <p>This paragraph is injected into the Card's default slot.</p>
  </Card>
</template>
```

Everything between the opening and closing `<Card>` tags replaces the `<slot />` element in the child's template.

## Fallback Content

You can define default content inside the `<slot>` tag. It renders when the parent does not provide anything for that slot.

```vue
<!-- SubmitButton.vue -->
<template>
  <button type="submit" class="btn">
    <slot>Submit</slot>
  </button>
</template>
```

```vue
<!-- Usage -->
<SubmitButton />                   <!-- renders "Submit" -->
<SubmitButton>Save</SubmitButton>  <!-- renders "Save" -->
```

Fallback content is useful for providing sensible defaults while keeping the component customizable.

## Named Slots

When a component needs multiple slot areas, use **named slots**. Each slot gets a `name` attribute, and the parent targets them with `v-slot` (or its shorthand `#`).

```vue
<!-- PageLayout.vue -->
<template>
  <div class="page">
    <header>
      <slot name="header" />
    </header>
    <main>
      <slot />  <!-- default slot -->
    </main>
    <footer>
      <slot name="footer" />
    </footer>
  </div>
</template>
```

```vue
<!-- Parent.vue -->
<template>
  <PageLayout>
    <template #header>
      <h1>My Application</h1>
    </template>

    <p>Main content goes into the default slot.</p>

    <template #footer>
      <p>&copy; 2026 My Company</p>
    </template>
  </PageLayout>
</template>
```

Key points:

- Named slots use `<template #slotName>` (shorthand for `<template v-slot:slotName>`).
- Content outside any `<template>` block goes to the default slot.
- A component can have any number of named slots plus one default slot.

## Scoped Slots

Sometimes the **child** component has data that the **parent** needs access to when rendering slot content. Scoped slots make this possible by letting the child pass data back up to the slot.

### Child Provides Data

```vue
<!-- UserList.vue -->
<script setup>
const users = [
  { id: 1, name: 'Alice', role: 'Admin' },
  { id: 2, name: 'Bob', role: 'Editor' },
  { id: 3, name: 'Carol', role: 'Viewer' }
]
</script>

<template>
  <ul>
    <li v-for="user in users" :key="user.id">
      <slot :user="user" :index="user.id" />
    </li>
  </ul>
</template>
```

### Parent Receives Data

```vue
<template>
  <UserList v-slot="{ user, index }">
    <span>{{ index }}. {{ user.name }} ({{ user.role }})</span>
  </UserList>
</template>
```

The child binds data to the `<slot>` element as attributes. The parent destructures those attributes in the `v-slot` directive.

### Scoped Slots with Named Slots

When combining named slots with scoped data, use the `#name="props"` syntax:

```vue
<!-- DataTable.vue -->
<template>
  <table>
    <thead>
      <tr>
        <slot name="header" />
      </tr>
    </thead>
    <tbody>
      <tr v-for="row in rows" :key="row.id">
        <slot name="row" :row="row" />
      </tr>
    </tbody>
    <tfoot>
      <tr>
        <slot name="footer" :total="rows.length" />
      </tr>
    </tfoot>
  </table>
</template>
```

```vue
<template>
  <DataTable :rows="products">
    <template #header>
      <th>Name</th>
      <th>Price</th>
    </template>

    <template #row="{ row }">
      <td>{{ row.name }}</td>
      <td>{{ row.price }}</td>
    </template>

    <template #footer="{ total }">
      <td colspan="2">Total items: {{ total }}</td>
    </template>
  </DataTable>
</template>
```

## Dynamic Slot Names

Slot names can be dynamic, computed at runtime using bracket syntax:

```vue
<script setup>
import { ref } from 'vue'

const currentSlot = ref('header')
</script>

<template>
  <PageLayout>
    <template #[currentSlot]>
      <p>This content targets the "{{ currentSlot }}" slot.</p>
    </template>
  </PageLayout>
</template>
```

Dynamic slot names are useful when building dynamic layouts or wizard-style interfaces where different content maps to different areas based on application state.

## Checking if a Slot Has Content

Inside a component, you can check whether a slot was provided using `$slots` in the template or `useSlots()` in `<script setup>`:

```vue
<script setup>
import { useSlots } from 'vue'

const slots = useSlots()
const hasHeader = !!slots.header
</script>

<template>
  <div class="card">
    <div v-if="$slots.header" class="card-header">
      <slot name="header" />
    </div>
    <div class="card-body">
      <slot />
    </div>
    <div v-if="$slots.footer" class="card-footer">
      <slot name="footer" />
    </div>
  </div>
</template>
```

This lets you conditionally render wrapper markup only when the parent actually provides content for a slot.

## The Renderless Component Pattern

One of the most powerful uses of scoped slots is the **renderless component** -- a component that handles logic but delegates all rendering to the parent via a scoped slot.

```vue
<!-- MouseTracker.vue -->
<script setup>
import { ref, onMounted, onUnmounted } from 'vue'

const x = ref(0)
const y = ref(0)

function update(event) {
  x.value = event.pageX
  y.value = event.pageY
}

onMounted(() => window.addEventListener('mousemove', update))
onUnmounted(() => window.removeEventListener('mousemove', update))
</script>

<template>
  <slot :x="x" :y="y" />
</template>
```

```vue
<!-- Parent.vue -->
<template>
  <MouseTracker v-slot="{ x, y }">
    <p>Mouse is at {{ x }}, {{ y }}</p>
  </MouseTracker>
</template>
```

The `MouseTracker` contains all the logic but renders nothing on its own. The parent decides how to display the data. This pattern is a precursor to composables and is still useful for components that need to wrap template content while providing behavior.

## Practical Example: Reusable List Component

```vue
<!-- FancyList.vue -->
<script setup>
import { ref, onMounted } from 'vue'

const props = defineProps({
  apiUrl: String
})

const items = ref([])
const loading = ref(true)

onMounted(async () => {
  const res = await fetch(props.apiUrl)
  items.value = await res.json()
  loading.value = false
})
</script>

<template>
  <div class="fancy-list">
    <slot v-if="loading" name="loading">
      <p>Loading...</p>
    </slot>

    <ul v-else>
      <li v-for="item in items" :key="item.id">
        <slot name="item" :item="item">
          {{ item.name }}
        </slot>
      </li>
    </ul>

    <slot v-if="!loading && items.length === 0" name="empty">
      <p>No items found.</p>
    </slot>
  </div>
</template>
```

```vue
<!-- Usage -->
<template>
  <FancyList api-url="/api/users">
    <template #loading>
      <div class="spinner">Fetching users...</div>
    </template>

    <template #item="{ item }">
      <div class="user-card">
        <strong>{{ item.name }}</strong>
        <span>{{ item.email }}</span>
      </div>
    </template>

    <template #empty>
      <p>No users found. Try adjusting your filters.</p>
    </template>
  </FancyList>
</template>
```

## Common Pitfalls

1. **Mixing default and named slot syntax.** When using named slots, default content must be in a `<template #default>` block or outside all named templates. Mixing them can cause confusion.

2. **Forgetting `<template>` wrapper.** Named slots require `<template>` tags. You cannot use `#header` directly on a plain element like `<div #header>`.

3. **Slot props scope.** Data from a scoped slot is only available inside that slot's template. You cannot access one slot's scoped data from another slot.

4. **Overusing renderless components.** Since Vue 3 introduced composables, many renderless component use cases are better served by `use*` functions. Prefer composables for pure logic; use renderless components when you need to wrap template structure.

## Summary

- **Default slots** let parents inject content into a single area.
- **Named slots** provide multiple content areas in one component.
- **Scoped slots** pass child data back to the parent for flexible rendering.
- **Dynamic slot names** let you target slots programmatically.
- Use `$slots` or `useSlots()` to conditionally render wrapper elements.
- The **renderless component** pattern separates logic from presentation using scoped slots.
