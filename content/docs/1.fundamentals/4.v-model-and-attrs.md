# v-model & Attribute Inheritance

In the previous lesson, you learned how props and events enable parent-child communication. This lesson builds on those concepts with two powerful features: **v-model on custom components** for two-way binding, and **attribute inheritance** for passing HTML attributes through component boundaries.

## v-model on Native Elements

You are likely already familiar with `v-model` on native form elements:

```vue
<template>
  <input v-model="searchQuery" />
  <!-- Equivalent to: -->
  <input :value="searchQuery" @input="searchQuery = $event.target.value" />
</template>
```

`v-model` is syntactic sugar that combines a `:value` binding with an `@input` listener. But its real power shines when you use it on **custom components**.

## v-model on Custom Components

When you use `v-model` on a component, Vue expands it into a prop and an event:

```vue
<!-- Parent.vue -->
<template>
  <CustomInput v-model="username" />
  <!-- Equivalent to: -->
  <CustomInput :modelValue="username" @update:modelValue="username = $event" />
</template>
```

The component receives a `modelValue` prop and emits an `update:modelValue` event to sync the value back.

### The Traditional Approach (Before `defineModel`)

Before Vue 3.4, you had to manually wire up the prop and event:

```vue
<!-- CustomInput.vue -->
<script setup>
const props = defineProps({
  modelValue: String
})

const emit = defineEmits(['update:modelValue'])

function onInput(event) {
  emit('update:modelValue', event.target.value)
}
</script>

<template>
  <input :value="modelValue" @input="onInput" />
</template>
```

This works, but it is verbose. You need a prop, an event declaration, and a handler function just to pass a value through.

## `defineModel()` -- The Modern Approach

Vue 3.4 introduced `defineModel()`, a compiler macro that dramatically simplifies two-way binding in custom components:

```vue
<!-- CustomInput.vue -->
<script setup>
const model = defineModel()
</script>

<template>
  <input v-model="model" />
</template>
```

That is the entire component. `defineModel()` returns a ref that:

- Automatically receives the value from the parent's `v-model`.
- Automatically emits `update:modelValue` when the ref is changed.
- Can be used directly with `v-model` on native elements inside the component.

### Using defineModel with Options

You can pass type and validation options to `defineModel()`:

```vue
<script setup>
const model = defineModel({
  type: String,
  required: true,
  default: ''
})
</script>
```

### TypeScript with defineModel

```vue
<script setup lang="ts">
const model = defineModel<string>({ required: true })
</script>
```

## Multiple v-model Bindings

A single component can support **multiple** v-model bindings by using named models. This is especially useful for form components that manage several values:

```vue
<!-- Parent.vue -->
<template>
  <UserForm
    v-model:firstName="first"
    v-model:lastName="last"
    v-model:email="emailAddress"
  />
</template>

<script setup>
import { ref } from 'vue'

const first = ref('Alice')
const last = ref('Johnson')
const emailAddress = ref('alice@example.com')
</script>
```

Inside the child component, use `defineModel()` with a name argument:

```vue
<!-- UserForm.vue -->
<script setup>
const firstName = defineModel('firstName')
const lastName = defineModel('lastName')
const email = defineModel('email')
</script>

<template>
  <form>
    <div>
      <label>First Name</label>
      <input v-model="firstName" />
    </div>
    <div>
      <label>Last Name</label>
      <input v-model="lastName" />
    </div>
    <div>
      <label>Email</label>
      <input v-model="email" type="email" />
    </div>
  </form>
</template>
```

Each named model maps to a separate prop (`firstName`, `lastName`, `email`) and event (`update:firstName`, etc.).

### Multiple v-model Without defineModel

For reference, here is how multiple v-model bindings work with the manual prop-and-event approach:

```vue
<script setup>
const props = defineProps({
  firstName: String,
  lastName: String
})

const emit = defineEmits(['update:firstName', 'update:lastName'])
</script>

<template>
  <input
    :value="firstName"
    @input="emit('update:firstName', $event.target.value)"
  />
  <input
    :value="lastName"
    @input="emit('update:lastName', $event.target.value)"
  />
</template>
```

The `defineModel` version is clearly more concise.

## v-model Modifiers

Vue's built-in modifiers (`.trim`, `.number`, `.lazy`) work on component v-model too. You can also create **custom modifiers**.

### Accessing Modifiers

`defineModel()` returns modifiers via its second element when destructured:

```vue
<!-- CustomInput.vue -->
<script setup>
const [model, modifiers] = defineModel({
  set(value) {
    // Apply the capitalize modifier
    if (modifiers.capitalize) {
      return value.charAt(0).toUpperCase() + value.slice(1)
    }
    return value
  }
})
</script>

<template>
  <input v-model="model" />
</template>
```

Usage in the parent:

```vue
<template>
  <CustomInput v-model.capitalize="name" />
</template>
```

## Attribute Inheritance (`$attrs`)

When you pass attributes to a component that are **not** declared as props or emits, they become **fallthrough attributes** and are accessible via `$attrs`.

```vue
<!-- Parent.vue -->
<template>
  <MyButton class="primary" id="save-btn" data-testid="save" />
</template>
```

If `MyButton` does not declare `class`, `id`, or `data-testid` as props, Vue automatically applies them to the component's root element:

```vue
<!-- MyButton.vue -->
<script setup>
// class, id, and data-testid are NOT declared as props
</script>

<template>
  <!-- These attributes are automatically applied to this root div -->
  <button>Save</button>
</template>

<!-- Rendered HTML: -->
<!-- <button class="primary" id="save-btn" data-testid="save">Save</button> -->
```

### What Falls Through?

The following are included in `$attrs`:

- HTML attributes (`class`, `style`, `id`, `data-*`, etc.)
- Event listeners (`@click`, `@focus`, etc.) that are not declared in `defineEmits`
- `aria-*` attributes

### Class and Style Merging

If the root element already has `class` or `style`, the fallthrough values are **merged**, not replaced:

```vue
<!-- MyButton.vue -->
<template>
  <button class="btn">Save</button>
</template>

<!-- Parent uses: <MyButton class="primary" /> -->
<!-- Rendered: <button class="btn primary">Save</button> -->
```

## Disabling Attribute Inheritance

Sometimes you do not want attributes to automatically land on the root element. For example, you might want to apply them to an inner element instead.

Disable the default behavior with `inheritAttrs: false`:

```vue
<script setup>
defineOptions({
  inheritAttrs: false
})
</script>

<template>
  <div class="input-wrapper">
    <!-- Manually bind $attrs to the input instead of the root div -->
    <label>{{ label }}</label>
    <input v-bind="$attrs" />
  </div>
</template>
```

Without `inheritAttrs: false`, the attributes would go to the outer `<div>`. With it disabled, you control exactly where they end up by using `v-bind="$attrs"`.

### Accessing $attrs in Script

In `<script setup>`, access attrs using the `useAttrs()` composable:

```vue
<script setup>
import { useAttrs } from 'vue'

const attrs = useAttrs()
console.log(attrs.class) // whatever class the parent passed
</script>
```

## Multi-Root Components and $attrs

If your component has **multiple root elements** (a Vue 3 fragment), Vue cannot automatically determine which element should receive the fallthrough attributes. You will see a runtime warning.

To fix this, explicitly bind `$attrs` to one of the root elements:

```vue
<script setup>
defineOptions({
  inheritAttrs: false
})
</script>

<template>
  <header>...</header>
  <main v-bind="$attrs">...</main>
  <footer>...</footer>
</template>
```

## Real-World Example: Wrapper Input Component

Here is a practical example that combines `defineModel`, `$attrs`, and `inheritAttrs: false` to build a reusable input wrapper:

```vue
<!-- FormInput.vue -->
<script setup>
defineOptions({
  inheritAttrs: false
})

const model = defineModel()

defineProps({
  label: String,
  error: String
})
</script>

<template>
  <div class="form-group">
    <label v-if="label" class="form-label">{{ label }}</label>
    <input
      v-model="model"
      v-bind="$attrs"
      class="form-input"
      :class="{ 'has-error': error }"
    />
    <p v-if="error" class="error-message">{{ error }}</p>
  </div>
</template>
```

Usage:

```vue
<template>
  <FormInput
    v-model="email"
    label="Email Address"
    type="email"
    placeholder="you@example.com"
    required
    :error="emailError"
  />
</template>
```

In this example:

- `label` and `error` are declared props -- they are used in the component's template.
- `type`, `placeholder`, and `required` are **not** declared as props -- they fall through to `$attrs` and are applied to the `<input>` element via `v-bind="$attrs"`.
- `v-model` is handled by `defineModel()`.

## Practical Tips

- **Use `defineModel()` for all new components** that need two-way binding. It is cleaner than the manual prop + emit approach.
- **Use named models** when a component needs to sync more than one value with the parent.
- **Set `inheritAttrs: false`** when building wrapper components where the root element is not the target for HTML attributes.
- **Always bind `$attrs` explicitly** on multi-root components to avoid warnings.
- **Think of `$attrs` as a passthrough mechanism.** It is especially useful for building design system components that wrap native elements.

## Common Pitfalls

| Pitfall | Solution |
|---|---|
| Using `defineModel()` in Vue versions before 3.4 | Upgrade to Vue 3.4+, or use the manual `modelValue` prop + `update:modelValue` event pattern |
| Forgetting `inheritAttrs: false` when manually binding `$attrs` | The attributes get applied twice -- once automatically to the root, once manually. Always disable inheritance when you use `v-bind="$attrs"` explicitly |
| Multi-root component with no explicit `$attrs` binding | Add `v-bind="$attrs"` to the element that should receive the fallthrough attributes |
| Declaring an attribute as a prop AND expecting it in `$attrs` | Once declared as a prop, it is consumed and will not appear in `$attrs` |
| Using `v-model` with a mismatched event name | Ensure your component emits `update:modelValue` (or `update:yourName` for named models) |

## Summary

`v-model` on custom components provides elegant two-way data binding, and `defineModel()` makes implementing it trivially simple. When you need multiple synced values, named models let you bind several v-model directives to one component. Meanwhile, `$attrs` and `inheritAttrs` give you fine-grained control over how HTML attributes flow through your component tree. Together, these features let you build flexible, reusable components that feel natural to use.
