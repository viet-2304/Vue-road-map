# Component Registration

Before you can use a component in a template, Vue needs to know about it. This process is called **component registration**. Understanding the different registration methods -- and when to use each -- is essential for building well-structured Vue applications.

## Global Registration

Global registration makes a component available throughout your entire application. You register it on the Vue application instance using `app.component()`:

```js
// main.js
import { createApp } from 'vue'
import App from './App.vue'
import BaseButton from './components/BaseButton.vue'
import BaseCard from './components/BaseCard.vue'

const app = createApp(App)

app.component('BaseButton', BaseButton)
app.component('BaseCard', BaseCard)

app.mount('#app')
```

After global registration, you can use `<BaseButton />` and `<BaseCard />` in the template of **any** component in your app without importing them again.

### When Global Registration Makes Sense

- **Base/UI components** used on almost every page (buttons, inputs, cards, modals).
- **Layout components** like headers, footers, and sidebars.

### Drawbacks of Global Registration

Global registration has some important downsides:

1. **No tree-shaking.** Globally registered components are included in the final bundle even if they are never used. This increases your bundle size unnecessarily.
2. **Implicit dependencies.** When a component uses a globally registered component, the dependency is not visible in the file. This makes the code harder to understand and maintain.
3. **Name collisions.** In large apps, global names can accidentally conflict.

For these reasons, **local registration is generally preferred**.

## Local Registration

With `<script setup>`, local registration is automatic. Simply import the component and use it in the template:

```vue
<script setup>
import UserAvatar from './UserAvatar.vue'
import UserBio from './UserBio.vue'
</script>

<template>
  <div class="profile">
    <UserAvatar />
    <UserBio />
  </div>
</template>
```

There is no explicit registration step -- the import itself is enough. This is one of the key benefits of `<script setup>`.

### Local Registration Without `<script setup>`

If you are using the Options API or a regular `<script>` block, you need to explicitly register imported components:

```vue
<script>
import UserAvatar from './UserAvatar.vue'
import UserBio from './UserBio.vue'

export default {
  components: {
    UserAvatar,
    UserBio
  }
}
</script>
```

Each property in the `components` object maps a template name to the imported component. With ES6 shorthand (`UserAvatar` instead of `UserAvatar: UserAvatar`), the name and value are the same.

## Naming Conventions

Vue supports two naming conventions for components in templates, and understanding them prevents confusion.

### PascalCase

```vue
<template>
  <UserProfile />
  <BaseButton label="Save" />
</template>
```

### kebab-case

```vue
<template>
  <user-profile />
  <base-button label="Save" />
</template>
```

Both work because Vue automatically resolves PascalCase component names to their kebab-case equivalents.

### Best Practices for Naming

| Context | Convention | Example |
|---|---|---|
| File names | PascalCase | `UserProfile.vue` |
| SFC templates | PascalCase | `<UserProfile />` |
| DOM templates (in HTML files) | kebab-case | `<user-profile>` |
| Component registration | PascalCase | `app.component('UserProfile', ...)` |

**Why PascalCase in SFC templates?**

- It visually distinguishes custom components from native HTML elements.
- It matches the JavaScript import name, making it easy to search your codebase.
- Self-closing tags (`<MyComponent />`) work naturally.

**Why kebab-case in DOM templates?**

HTML is case-insensitive, so browsers parse `<UserProfile>` as `<userprofile>`. When using Vue directly in an HTML file (without a build step), you must use kebab-case.

## Multi-Word Component Names

Vue strongly recommends that component names always consist of **multiple words**. This avoids conflicts with existing and future HTML elements, which are always single words (like `<header>`, `<table>`, `<input>`).

```
Bad:   Button.vue, Card.vue, Table.vue
Good:  BaseButton.vue, AppCard.vue, DataTable.vue
```

Common prefixes for base components include `Base`, `App`, or `V`:

```
BaseButton.vue
BaseInput.vue
BaseCard.vue
```

## Auto-Import in Nuxt

If you are building with **Nuxt 3**, component registration is handled for you automatically. Nuxt scans the `components/` directory and auto-imports every component found there.

```
components/
  AppHeader.vue
  AppFooter.vue
  ui/
    BaseButton.vue
    BaseCard.vue
```

You can use these directly in any page or component without importing them:

```vue
<template>
  <AppHeader />
  <main>
    <UiBaseButton label="Click me" />
  </main>
  <AppFooter />
</template>
```

### How Nuxt Resolves Nested Names

By default, Nuxt uses the **directory path** as part of the component name. A file at `components/ui/BaseButton.vue` is auto-imported as `<UiBaseButton />`.

You can customize this behavior in `nuxt.config.ts`:

```ts
// nuxt.config.ts
export default defineNuxtConfig({
  components: [
    {
      path: '~/components',
      pathPrefix: false  // removes directory prefix from names
    }
  ]
})
```

With `pathPrefix: false`, the same file becomes just `<BaseButton />`.

### Benefits of Nuxt Auto-Import

- **Zero boilerplate.** No import statements needed.
- **Automatic tree-shaking.** Only components actually used in templates are included in the bundle.
- **TypeScript support.** Nuxt generates type declarations so you get full autocompletion in your editor.

## Dynamic Components

Sometimes you need to switch between components dynamically based on data or user interaction. Vue provides the built-in `<component>` element with the `:is` attribute for this:

```vue
<script setup>
import { ref } from 'vue'
import TabHome from './TabHome.vue'
import TabSettings from './TabSettings.vue'
import TabProfile from './TabProfile.vue'

const tabs = {
  home: TabHome,
  settings: TabSettings,
  profile: TabProfile
}

const currentTab = ref('home')
</script>

<template>
  <div class="tabs">
    <button
      v-for="(_, name) in tabs"
      :key="name"
      :class="{ active: currentTab === name }"
      @click="currentTab = name"
    >
      {{ name }}
    </button>
  </div>

  <component :is="tabs[currentTab]" />
</template>
```

The `:is` attribute accepts:

- A **component definition object** (imported component).
- A **string** name of a registered component.

### Keeping Dynamic Components Alive

By default, when you switch away from a dynamic component, it is destroyed and its state is lost. Wrap it with `<KeepAlive>` to preserve its state:

```vue
<KeepAlive>
  <component :is="tabs[currentTab]" />
</KeepAlive>
```

This caches the component instances so that form inputs, scroll positions, and other state survive tab switches.

## Practical Tips

- **Prefer local registration** (or Nuxt auto-import) over global registration. This makes dependencies explicit and keeps bundles lean.
- **Be consistent with naming.** Pick PascalCase for SFC templates and stick with it across your project.
- **Use multi-word names** for every component to avoid HTML element conflicts.
- **Leverage Nuxt auto-import** when working in a Nuxt project -- it eliminates registration boilerplate entirely.

## Common Pitfalls

| Pitfall | Solution |
|---|---|
| Component renders as empty or unknown HTML element | Check that you imported and registered it (or that Nuxt auto-import can find it) |
| Name collision with HTML elements (`<Header>`, `<Footer>`) | Use multi-word names: `AppHeader`, `AppFooter` |
| Globally registering dozens of components | Switch to local registration or Nuxt auto-import for better tree-shaking |
| Forgetting to use `:is` with dynamic components | `<component is="TabHome" />` treats the string as a registered component name; use `:is="TabHome"` for imported components |

## Summary

Component registration tells Vue which components are available in a given template. Global registration is convenient but comes at the cost of bundle size and code clarity. Local registration with `<script setup>` is concise and explicit. Nuxt takes this further with automatic imports from the `components/` directory. No matter which approach you use, following consistent naming conventions keeps your codebase readable and collision-free.
