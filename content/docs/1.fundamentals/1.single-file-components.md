# Single File Components

Single File Components (SFCs) are the cornerstone of Vue.js development. They let you define a component's template, logic, and styles in a single `.vue` file, keeping everything related to one component neatly organized in one place.

## What Is a Single File Component?

An SFC is a file with the `.vue` extension that encapsulates three concerns of a UI component:

- **Template** -- the HTML structure
- **Script** -- the JavaScript logic
- **Style** -- the CSS styling

This co-location of concerns is one of Vue's most loved features. Instead of splitting your code across separate HTML, JS, and CSS files, you group them by component. This makes components easier to reason about, maintain, and refactor.

## The `.vue` File Structure

Every SFC follows the same basic structure with up to three top-level blocks:

```vue
<template>
  <div class="greeting">
    <h1>{{ message }}</h1>
    <p>Welcome to Vue.js</p>
  </div>
</template>

<script setup>
import { ref } from 'vue'

const message = ref('Hello, World!')
</script>

<style scoped>
.greeting {
  text-align: center;
  padding: 2rem;
}

h1 {
  color: #42b883;
}
</style>
```

Let's break down each block.

### The `<template>` Block

The template block contains the HTML markup for your component. Vue uses an HTML-based template syntax that allows you to declaratively bind the rendered DOM to the underlying component data.

```vue
<template>
  <div>
    <h2>{{ title }}</h2>
    <button @click="handleClick">Click me</button>
    <ul>
      <li v-for="item in items" :key="item.id">
        {{ item.name }}
      </li>
    </ul>
  </div>
</template>
```

Key points about the template block:

- Each component must have exactly **one root element** (though Vue 3 supports multiple root elements via fragments).
- You can use Vue directives like `v-if`, `v-for`, `v-bind`, and `v-on`.
- Expressions inside `{{ }}` are evaluated as JavaScript.

### The `<script setup>` Block

The `<script setup>` syntax is a compile-time syntactic sugar that makes Composition API code inside SFCs more concise. It is the recommended approach for Vue 3 components.

```vue
<script setup>
import { ref, computed } from 'vue'
import MyButton from './MyButton.vue'

// Reactive state
const count = ref(0)

// Computed property
const doubled = computed(() => count.value * 2)

// Methods
function increment() {
  count.value++
}
</script>
```

With `<script setup>`, several things happen automatically:

1. **Top-level bindings are exposed to the template.** Any variable, function, or import declared at the top level is directly usable in the template -- no need for a `return` statement.
2. **Imported components are automatically registered.** When you import a component, it becomes available in the template without manual registration.
3. **Better TypeScript support.** Type inference works more naturally.
4. **Better runtime performance.** The compiled code can access variables directly without a proxy layer.

### Comparison: `<script setup>` vs. Options API

To appreciate `<script setup>`, compare it with the Options API approach:

```vue
<!-- Options API (the older pattern) -->
<script>
import MyButton from './MyButton.vue'

export default {
  components: { MyButton },
  data() {
    return {
      count: 0
    }
  },
  computed: {
    doubled() {
      return this.count * 2
    }
  },
  methods: {
    increment() {
      this.count++
    }
  }
}
</script>
```

```vue
<!-- Composition API with <script setup> (recommended) -->
<script setup>
import { ref, computed } from 'vue'
import MyButton from './MyButton.vue'

const count = ref(0)
const doubled = computed(() => count.value * 2)

function increment() {
  count.value++
}
</script>
```

The `<script setup>` version is shorter, has less boilerplate, and keeps related logic together.

### The `<style>` Block

The style block contains CSS for the component. You can include multiple style blocks if needed.

```vue
<style scoped>
.card {
  border: 1px solid #e2e8f0;
  border-radius: 8px;
  padding: 1.5rem;
}

.card h2 {
  margin-top: 0;
}
</style>
```

## Understanding `<style scoped>`

By default, styles in an SFC are **global** -- they affect every element on the page. Adding the `scoped` attribute restricts the styles to the current component only.

```vue
<style scoped>
/* This h1 rule ONLY applies to <h1> elements in this component */
h1 {
  color: red;
}
</style>
```

Under the hood, Vue achieves this by adding a unique data attribute (like `data-v-7ba5bd90`) to each element in the component and transforming your CSS selectors to include that attribute.

### Scoped Style Caveats

There are a few things to keep in mind when using scoped styles:

1. **Child component root elements** are affected by both the parent's scoped CSS and the child's scoped CSS. This is by design so the parent can style the child's root element for layout purposes.

2. **Deep selectors.** If you need to affect child component internals from a parent, use the `:deep()` pseudo-class:

```vue
<style scoped>
/* Targets .inner elements inside child components */
.parent :deep(.inner) {
  color: blue;
}
</style>
```

3. **Slotted selectors.** To target content passed via slots, use `:slotted()`:

```vue
<style scoped>
:slotted(.slot-content) {
  font-weight: bold;
}
</style>
```

4. **Global selectors within scoped blocks.** Use `:global()` to apply a single rule globally:

```vue
<style scoped>
:global(.some-global-class) {
  margin: 0;
}
</style>
```

## Using CSS Preprocessors

SFCs support CSS preprocessors out of the box. Just add a `lang` attribute:

```vue
<style scoped lang="scss">
$primary: #42b883;

.card {
  border: 1px solid lighten($primary, 30%);

  &:hover {
    border-color: $primary;
  }
}
</style>
```

You can also use `lang="less"`, `lang="stylus"`, or any preprocessor with the appropriate dependency installed.

## When to Use SFCs

SFCs are ideal for virtually all Vue development scenarios:

- **Application development** -- SFCs are the standard way to build Vue apps with build tools like Vite or Nuxt.
- **Reusable component libraries** -- SFCs keep each component self-contained and easy to distribute.
- **Prototyping** -- The co-location of template, logic, and style makes rapid iteration straightforward.

The only time you might skip SFCs is when embedding Vue into an existing page without a build step, using inline templates or render functions.

## Practical Tips

- **One component per file.** Always define a single component per `.vue` file. This keeps things focused and predictable.
- **Use `<script setup>` by default.** Unless you have a specific reason to use the Options API, prefer `<script setup>` for all new components.
- **Always scope your styles.** Add `scoped` to your style blocks to prevent accidental style leaks. Use `:deep()` only when truly necessary.
- **Name your files with PascalCase.** For example, `UserProfile.vue`, `NavBar.vue`. This aligns with how you use them in templates.
- **Keep templates readable.** If your template grows beyond 50-60 lines, consider breaking it into smaller child components.

## Common Pitfalls

| Pitfall | Solution |
|---|---|
| Forgetting `scoped` and leaking styles globally | Always add `scoped` unless you intentionally want global styles |
| Using `<script>` without `setup` and wondering why top-level variables are not available in the template | Add the `setup` attribute: `<script setup>` |
| Nesting `<template>` blocks (invalid) | Only one `<template>` block per SFC at the top level |
| Trying to use SFCs without a build tool | SFCs require compilation -- use Vite, Nuxt, or Vue CLI |

## Summary

Single File Components bring together HTML, JavaScript, and CSS in one `.vue` file, giving you a clean, maintainable way to build Vue applications. The `<script setup>` syntax reduces boilerplate and improves developer experience, while `<style scoped>` keeps your styles isolated. These features make SFCs the foundation you will build on throughout the rest of this course.
