# Props & Events

Components need to communicate with each other. In Vue, the primary mechanism for this is **props down, events up**: a parent passes data to a child via props, and a child notifies the parent of changes via events. This one-way data flow keeps your application predictable and easy to debug.

## What Are Props?

Props (short for "properties") are custom attributes you can register on a component. When a parent passes a value to a prop, it becomes a property on the child component instance.

```vue
<!-- Parent.vue -->
<template>
  <UserCard name="Alice" :age="28" role="Developer" />
</template>
```

```vue
<!-- UserCard.vue -->
<script setup>
const props = defineProps({
  name: String,
  age: Number,
  role: String
})
</script>

<template>
  <div class="card">
    <h2>{{ name }}</h2>
    <p>Age: {{ age }}</p>
    <p>Role: {{ role }}</p>
  </div>
</template>
```

Notice the `:age="28"` syntax. The colon (shorthand for `v-bind`) tells Vue to evaluate `28` as a JavaScript expression (a number), not a string.

## Defining Props with `defineProps`

In `<script setup>`, you declare props using the `defineProps()` compiler macro. It does not need to be imported -- Vue's compiler handles it automatically.

### Runtime Declaration

Pass an object where keys are prop names and values are constructor types:

```vue
<script setup>
const props = defineProps({
  title: String,
  count: Number,
  isActive: Boolean,
  tags: Array,
  config: Object
})
</script>
```

### Type-Based Declaration (TypeScript)

If you are using TypeScript, you can declare props with a type interface for better type safety:

```vue
<script setup lang="ts">
interface Props {
  title: string
  count: number
  isActive: boolean
  tags: string[]
  config: Record<string, unknown>
}

const props = defineProps<Props>()
</script>
```

You cannot mix runtime and type-based declarations -- use one or the other.

## Prop Validation

Vue provides a rich validation system for props. Instead of just specifying the type, pass an object with validation options:

```vue
<script setup>
const props = defineProps({
  // Basic type check
  title: String,

  // Multiple possible types
  id: [String, Number],

  // Required prop
  username: {
    type: String,
    required: true
  },

  // Prop with a default value
  role: {
    type: String,
    default: 'viewer'
  },

  // Array/Object defaults must use a factory function
  items: {
    type: Array,
    default: () => []
  },

  // Custom validator function
  status: {
    type: String,
    validator(value) {
      return ['active', 'inactive', 'pending'].includes(value)
    }
  }
})
</script>
```

### Defaults with TypeScript

When using the type-based declaration, use `withDefaults` to set defaults:

```vue
<script setup lang="ts">
interface Props {
  title: string
  role?: string
  items?: string[]
}

const props = withDefaults(defineProps<Props>(), {
  role: 'viewer',
  items: () => []
})
</script>
```

### Boolean Props

Boolean props have special behavior. When a boolean prop is present on a tag without a value, it is treated as `true`:

```vue
<!-- These are equivalent -->
<Modal visible />
<Modal :visible="true" />
```

And when omitted entirely, it defaults to `false` (unless you specify a different default).

## One-Way Data Flow

Props follow a **one-way-down** binding. When the parent property updates, the new value flows down to the child, but not the other way around. You should **never mutate a prop directly** inside a child component.

```vue
<script setup>
const props = defineProps({
  initialCount: Number
})

// BAD -- do not mutate props directly
// props.initialCount++

// GOOD -- create a local copy if you need to modify the value
import { ref } from 'vue'
const localCount = ref(props.initialCount)

function increment() {
  localCount.value++
}
</script>
```

If you need a computed transformation of a prop, use a computed property:

```vue
<script setup>
import { computed } from 'vue'

const props = defineProps({
  size: String
})

const normalizedSize = computed(() => props.size.trim().toLowerCase())
</script>
```

## What Are Events?

When a child component needs to communicate something back to its parent -- like a button click, a form submission, or a data change -- it **emits an event**. The parent listens for that event using `v-on` (or the `@` shorthand).

## Defining Events with `defineEmits`

In `<script setup>`, declare the events your component can emit using `defineEmits()`:

```vue
<!-- SearchInput.vue -->
<script setup>
const emit = defineEmits(['search', 'clear'])

function handleSearch(query) {
  emit('search', query)
}

function handleClear() {
  emit('clear')
}
</script>

<template>
  <div class="search-input">
    <input
      type="text"
      @keyup.enter="handleSearch($event.target.value)"
      placeholder="Search..."
    />
    <button @click="handleClear">Clear</button>
  </div>
</template>
```

The parent listens for these events:

```vue
<!-- Parent.vue -->
<template>
  <SearchInput
    @search="performSearch"
    @clear="clearResults"
  />
</template>

<script setup>
function performSearch(query) {
  console.log('Searching for:', query)
}

function clearResults() {
  console.log('Results cleared')
}
</script>
```

### Event Validation

Just like props, you can validate emitted events by providing a validation function:

```vue
<script setup>
const emit = defineEmits({
  // No validation
  clear: null,

  // Validate the search event payload
  search: (query) => {
    if (!query) {
      console.warn('Search event requires a query string')
      return false
    }
    return true
  }
})
</script>
```

### TypeScript Event Declarations

With TypeScript, use the type-based syntax for precise typing:

```vue
<script setup lang="ts">
const emit = defineEmits<{
  search: [query: string]
  clear: []
  update: [id: number, value: string]
}>()
</script>
```

This gives you full type checking on both the `emit()` calls inside the component and the `@search`, `@clear`, `@update` handlers in the parent.

## Putting It All Together

Here is a complete example showing a parent-child communication pattern:

```vue
<!-- TodoItem.vue -->
<script setup>
defineProps({
  todo: {
    type: Object,
    required: true
  }
})

const emit = defineEmits(['toggle', 'delete'])
</script>

<template>
  <li class="todo-item">
    <input
      type="checkbox"
      :checked="todo.done"
      @change="emit('toggle', todo.id)"
    />
    <span :class="{ done: todo.done }">{{ todo.text }}</span>
    <button @click="emit('delete', todo.id)">Delete</button>
  </li>
</template>
```

```vue
<!-- TodoList.vue -->
<script setup>
import { ref } from 'vue'
import TodoItem from './TodoItem.vue'

const todos = ref([
  { id: 1, text: 'Learn Vue props', done: false },
  { id: 2, text: 'Learn Vue events', done: false },
  { id: 3, text: 'Build something cool', done: false }
])

function toggleTodo(id) {
  const todo = todos.value.find(t => t.id === id)
  if (todo) todo.done = !todo.done
}

function deleteTodo(id) {
  todos.value = todos.value.filter(t => t.id !== id)
}
</script>

<template>
  <ul>
    <TodoItem
      v-for="todo in todos"
      :key="todo.id"
      :todo="todo"
      @toggle="toggleTodo"
      @delete="deleteTodo"
    />
  </ul>
</template>
```

This pattern clearly shows the data flowing **down** through the `:todo` prop and actions flowing **up** through the `@toggle` and `@delete` events.

## Practical Tips

- **Always declare your props and events.** Even though Vue allows undeclared props and events, explicit declarations serve as documentation and enable validation.
- **Use the type-based syntax in TypeScript projects.** It provides the best developer experience with full type inference.
- **Keep event names descriptive.** Use names like `update:title`, `submit`, `delete` rather than vague names like `action` or `change`.
- **Prefer kebab-case for event names in templates.** Write `@item-selected` instead of `@itemSelected`. Vue automatically handles the conversion.
- **Do not emit events that shadow native HTML events** (like `click` or `input`) unless you intentionally want to override them.

## Common Pitfalls

| Pitfall | Solution |
|---|---|
| Mutating props directly inside a child component | Create a local `ref` copy or emit an event asking the parent to update |
| Passing a string when a number is expected (`count="5"`) | Use `v-bind` to pass the value as an expression: `:count="5"` |
| Forgetting to use a factory function for object/array defaults | Always use `default: () => []` or `default: () => ({})` for non-primitive defaults |
| Mixing runtime and type-based `defineProps` | Choose one approach and stick with it within a component |
| Not listening for an emitted event in the parent | Double-check the event name matches between `emit('myEvent')` and `@my-event="handler"` |

## Summary

Props and events form the backbone of parent-child communication in Vue. Props flow data downward from parent to child, while events send messages upward from child to parent. The `defineProps` and `defineEmits` macros make declaring these interfaces clean and type-safe. By respecting one-way data flow and validating your props, you build components that are predictable, reusable, and easy to maintain.
