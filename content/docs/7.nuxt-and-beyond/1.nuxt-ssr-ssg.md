# Nuxt SSR/SSG

Modern web applications need to balance performance, SEO, and interactivity. Nuxt 3 provides multiple rendering strategies out of the box, letting you choose the right approach for each page of your application.

## Understanding Rendering Modes

Before diving into Nuxt-specific features, let's clarify the three main rendering strategies.

### Client-Side Rendering (CSR)

The traditional SPA approach. The server sends a minimal HTML shell, and JavaScript builds the page in the browser.

```
Browser requests page → Server sends empty HTML + JS bundle → Browser downloads JS → JS renders the page
```

**Pros:** Simple deployment, rich interactivity
**Cons:** Poor SEO, slow initial load, blank page until JS loads

### Server-Side Rendering (SSR)

The server generates the full HTML for each request. The browser receives a complete page, then Vue "hydrates" it to make it interactive.

```
Browser requests page → Server runs Vue, generates HTML → Browser shows HTML immediately → JS hydrates the page
```

**Pros:** Great SEO, fast first paint, works without JS
**Cons:** Higher server load, TTFB depends on server speed

### Static Site Generation (SSG)

Pages are pre-rendered at build time. The resulting HTML files are served from a CDN with no server needed at runtime.

```
Build step generates HTML for every route → CDN serves pre-built HTML → Browser hydrates the page
```

**Pros:** Fastest possible load times, cheapest hosting, excellent SEO
**Cons:** Stale content until rebuild, not suitable for highly dynamic pages

## Configuring Rendering in Nuxt

Nuxt defaults to **universal rendering** (SSR). You can change this globally in `nuxt.config.ts`:

```ts
// nuxt.config.ts
export default defineNuxtConfig({
  // Full SSR (default)
  ssr: true,

  // Or pure client-side SPA
  // ssr: false,
})
```

## Static Site Generation with `nuxi generate`

To pre-render your entire site as static HTML:

```bash
# Generate static files
npx nuxi generate

# Preview the generated site
npx nuxi preview
```

Nuxt crawls your application starting from `/`, follows all links, and generates an HTML file for each discovered route. The output lands in `.output/public/`.

### Pre-rendering Specific Routes

You can tell Nuxt to pre-render specific routes even in SSR mode:

```ts
// nuxt.config.ts
export default defineNuxtConfig({
  nitro: {
    prerender: {
      routes: ['/about', '/contact'],
      // Crawl all links starting from these pages
      crawlLinks: true,
    },
  },
})
```

## Hybrid Rendering with routeRules

This is where Nuxt truly shines. With `routeRules`, you can assign **different rendering strategies per route** in a single application:

```ts
// nuxt.config.ts
export default defineNuxtConfig({
  routeRules: {
    // Static pages - generated at build time
    '/': { prerender: true },
    '/about': { prerender: true },

    // SPA mode - no server rendering
    '/admin/**': { ssr: false },

    // ISR - regenerate in the background every 60 seconds
    '/blog/**': { isr: 60 },

    // SSR with caching - cached for 10 minutes
    '/api/**': {
      cache: { maxAge: 60 * 10 },
    },

    // SWR (stale-while-revalidate)
    '/products/**': {
      swr: 3600, // Serve stale for up to 1 hour while revalidating
    },

    // Redirect rules
    '/old-page': { redirect: '/new-page' },

    // Add custom headers
    '/assets/**': {
      headers: {
        'cache-control': 'public, max-age=31536000, immutable',
      },
    },
  },
})
```

### Incremental Static Regeneration (ISR)

ISR combines the benefits of SSG and SSR. Pages are pre-rendered but can be refreshed on a schedule:

```ts
// nuxt.config.ts
export default defineNuxtConfig({
  routeRules: {
    // Rebuild this page at most once every 60 seconds
    '/blog/**': { isr: 60 },

    // Pre-render once, never revalidate (true = build-time only)
    '/docs/**': { isr: true },
  },
})
```

When a user requests `/blog/my-post`:
1. If a cached version exists and is fresh, serve it instantly
2. If stale, serve the cached version AND regenerate in the background
3. The next visitor gets the freshly generated page

## Payload Extraction

When Nuxt pre-renders a page, it can also extract the data payload separately. This allows client-side navigation to fetch only the JSON data instead of a full HTML page:

```ts
// nuxt.config.ts
export default defineNuxtConfig({
  experimental: {
    payloadExtraction: true,
  },
})
```

With payload extraction enabled, pre-rendered pages produce:
- `/about/index.html` — the full HTML page
- `/about/_payload.json` — the serialized data

When navigating client-side from another page to `/about`, Nuxt fetches only `_payload.json` instead of the full HTML.

## Data Fetching and Rendering

How you fetch data interacts directly with your rendering strategy:

```vue
<script setup lang="ts">
// useAsyncData works with SSR - data is fetched server-side
// and serialized into the HTML payload
const { data: posts, pending, error } = await useAsyncData(
  'posts',
  () => $fetch('/api/posts')
)

// useFetch is a convenience wrapper around useAsyncData + $fetch
const { data: user } = await useFetch('/api/user/me')
</script>

<template>
  <div v-if="pending">Loading...</div>
  <div v-else-if="error">Error: {{ error.message }}</div>
  <div v-else>
    <article v-for="post in posts" :key="post.id">
      <h2>{{ post.title }}</h2>
    </article>
  </div>
</template>
```

### The Hydration Mismatch Problem

When using SSR, the HTML rendered on the server must match what Vue produces on the client. Mismatches cause hydration errors:

```vue
<script setup lang="ts">
// BAD - different value on server vs client
const timestamp = ref(Date.now())

// GOOD - use useAsyncData so the value is consistent
const { data: timestamp } = await useAsyncData('time', () => {
  return Date.now()
})
</script>
```

## Choosing the Right Strategy

| Strategy | Best For | Trade-off |
|----------|----------|-----------|
| **SSR** | Dynamic, personalized pages | Higher server costs |
| **SSG** | Marketing pages, docs, blogs | Stale until rebuild |
| **ISR** | Content that updates periodically | Slight staleness window |
| **SWR** | API responses, product pages | Complexity |
| **CSR** | Admin dashboards, authenticated apps | No SEO |

## Practical Tips

1. **Start with SSR as the default** and opt specific routes into other modes using `routeRules`
2. **Pre-render your marketing pages** — they change rarely and benefit most from static delivery
3. **Use ISR for blog content** — fresh enough for most readers without rebuilding the entire site
4. **Disable SSR for admin panels** — they don't need SEO and benefit from SPA behavior
5. **Test with `nuxi preview`** — it simulates the production server locally so you can verify rendering behavior

## Common Pitfalls

- **Using `window` or `document` in SSR**: These don't exist on the server. Guard with `if (import.meta.client)` or use the `<ClientOnly>` component.
- **Forgetting to `await` data fetching**: If you don't `await` `useAsyncData` or `useFetch` at the top level of `<script setup>`, the page may render before data is ready.
- **Overusing SSG for dynamic routes**: If you have thousands of product pages that change frequently, ISR or SSR with caching is a better fit.

## Summary

- Nuxt supports **SSR**, **SSG**, **ISR**, **SWR**, and **CSR** out of the box
- Use `routeRules` to assign different strategies per route in a single app
- `nuxi generate` pre-renders your entire site as static HTML
- Payload extraction optimizes client-side navigation for pre-rendered pages
- Choose your rendering strategy based on content freshness needs, SEO requirements, and server budget
