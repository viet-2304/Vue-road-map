# Reactivity: ref & reactive

Vue's reactivity system is what makes the framework "reactive" — when data changes, the UI updates automatically. The Composition API gives you two main tools: `ref()` and `reactive()`.

## ref()

`ref()` wraps a single value in a reactive container. Access the value via `.value` in JavaScript, but in templates Vue unwraps it automatically.

```vue
<script setup lang="ts">
import { ref } from 'vue'

const count = ref(0)

function increment() {
  count.value++ // .value required in <script>
}
</script>

<template>
  <!-- No .value needed in template -->
  <button @click="increment">Count: {{ count }}</button>
</template>
```

### When to use ref

- Primitives: `string`, `number`, `boolean`
- When you need to reassign the entire value
- When you want a single reactive variable

## reactive()

`reactive()` makes an entire object deeply reactive. No `.value` needed, but you **cannot reassign** the variable itself.

```vue
<script setup lang="ts">
import { reactive } from 'vue'

const state = reactive({
  count: 0,
  name: 'Vue',
})

function increment() {
  state.count++ // direct access, no .value
}
</script>

<template>
  <p>{{ state.name }}: {{ state.count }}</p>
  <button @click="increment">+1</button>
</template>
```

### When to use reactive

- Objects with multiple related properties
- When you want a "state object" pattern
- When nested reactivity is needed

## ref vs reactive — Key Differences

| Feature | `ref()` | `reactive()` |
|---|---|---|
| Works with primitives | Yes | No |
| Requires `.value` | Yes (in JS) | No |
| Can reassign | Yes (`count.value = 5`) | No (loses reactivity) |
| Destructuring | Safe with `.value` | **Loses reactivity** |
| TypeScript inference | `Ref<T>` | The object type itself |

## The Destructuring Trap

One of the most common mistakes with `reactive()`:

```ts
const state = reactive({ count: 0, name: 'Vue' })

// BAD — loses reactivity!
const { count, name } = state

// GOOD — use toRefs()
const { count, name } = toRefs(state)
// count and name are now Ref<number> and Ref<string>
```

## Utility Functions

Vue provides helper functions for working with refs:

```ts
import { ref, toRef, toRefs, unref, isRef } from 'vue'

const state = reactive({ count: 0, name: 'Vue' })

// toRef: create a ref linked to a reactive property
const countRef = toRef(state, 'count')

// toRefs: convert all properties to refs
const { count, name } = toRefs(state)

// unref: unwrap a ref (returns .value or the value itself)
const val = unref(count) // 0

// isRef: check if a value is a ref
isRef(count) // true
isRef(42)    // false
```

## Best Practices

1. **Default to `ref()`** — it works with everything and avoids the destructuring trap
2. **Use `reactive()` for state objects** — when you have a group of related properties
3. **Never reassign a `reactive()` variable** — use `Object.assign()` instead
4. **Use `toRefs()` when destructuring** — preserves reactivity
5. **Use `shallowRef()` for large objects** — avoids deep reactivity overhead

```ts
// Instead of reassigning:
// state = { count: 1, name: 'Nuxt' } // BAD

// Do this:
Object.assign(state, { count: 1, name: 'Nuxt' }) // GOOD
```

## Summary

- `ref()` for single values — requires `.value` in JS
- `reactive()` for objects — no `.value` but can't reassign
- `toRefs()` / `toRef()` to safely destructure reactive objects
- Default to `ref()` unless you have a good reason for `reactive()`
